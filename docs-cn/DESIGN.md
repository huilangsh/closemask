# CloseMask 设计文档

## 系统架构

CloseMask 采用分层架构设计，将 PII 检测与代理功能分离，确保可维护性和可扩展性。

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                     应用层                              │
│  (Agent Applications, Client SDKs, Direct API Calls)   │
└────────────────────┬────────────────────────────────────┘
                     │ HTTP/SSE
                     ▼
┌─────────────────────────────────────────────────────────┐
│                  CloseMask 代理层                        │
├─────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  HTTP Server │  │  Stream      │  │  Session     │  │
│  │              │  │  Processor   │  │  Manager     │  │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  │
└─────────┼──────────────────┼──────────────────┼─────────┘
          │                  │                  │
          ▼                  ▼                  ▼
┌─────────────────────────────────────────────────────────┐
│                  业务逻辑层                              │
├─────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  Tool Call   │  │  Placeholder │  │  PII         │  │
│  │  Handler     │  │  Manager     │  │  Processor   │  │

                   │   OneAIFW       │
                   │  (PII 检测引擎) │
                   └─────────────────┘
```

## 核心组件

### 1. HTTP 服务器

**职责**：
- 接收 OpenAI 兼容的 HTTP 请求
- 路由请求到相应处理器
- 处理健康检查和指标端点

**关键特性**：
- 支持 POST /v1/chat/completions（流式和非流式）
- GET /health 健康检查
- GET /metrics Prometheus 指标
- 多提供商支持（通过 OpenAI 兼容协议）

**LLM 提供商兼容性**：

CloseMask 采用 OpenAI 兼容协议（`/v1/chat/completions`）与上游 LLM 通信，这种设计确保了广泛的兼容性：

| 提供商类型 | 接入方式 | 示例 `llm_url` |
|-----------|---------|---------------|
| OpenAI | 直接对接 | `https://api.openai.com/v1` |
| Azure OpenAI | 直接对接 | `https://{resource}.openai.azure.com/...` |
| Ollama | 直接对接 | `http://localhost:11434` |
| Groq | 直接对接 | `https://api.groq.com/openai/v1` |
| DeepSeek | 直接对接 | `https://api.deepseek.com/v1` |
| 智谱 AI | 直接对接 | `https://open.bigmodel.cn/api/paas/v4` |
| Anthropic Claude | 通过代理 | `http://localhost:3000/v1`（one-api） |
| 其他兼容提供商 | 直接对接 | 对应 base URL |

对于 Anthropic Claude，需要使用 OpenAI 兼容代理层（如 [one-api](https://github.com/songquanpeng/one-api)、[LiteLLM](https://github.com/BerriAI/litellm) 或 [OpenRouter](https://openrouter.ai/)），因为 Anthropic 原生 API 使用不同的协议格式（`/v1/messages`）。代理层负责协议转换，CloseMask 专注于 PII 保护。

### 2. 流式处理器（Stream Processor）

**职责**：
- 处理 SSE（Server-Sent Events）流式响应
- 在每个数据块中还原 PII 占位符
- 维护流式响应的连续性

**挑战与解决方案**：
- **挑战**：PII 占位符可能跨多个数据块
- **解决方案**：使用流缓冲和智能重组

### 3. 会话管理器（Session Manager）

**职责**：
- 管理多轮对话的会话状态
- 存储 PII 占位符映射关系
- 提供会话隔离和过期机制

**数据结构**：
```go
type Session struct {
    ID        string
    CreatedAt time.Time
    ExpiresAt time.Time
    Placeholders map[string]string  // 占位符 -> 原始值
}

type SessionStore interface {
    Create(sessionID string) (*Session, error)
    Get(sessionID string) (*Session, error)
    Update(sessionID string, placeholders map[string]string) error
    Delete(sessionID string) error
}
```

**存储策略**：
- **内存存储**：高性能，适合低并发
- **Redis 存储**：分布式，适合高并发
- **分层存储**：内存 + Redis 持久化

### 4. 工具调用处理器（Tool Call Handler）

**职责**：
- 检测和解析工具调用请求
- 遮罩工具调用参数中的 PII
- 还原工具结果中的 PII

**工作流程**：
```
1. 检测到 tool_calls
2. 解析工具参数
3. 调用 PII 检测器遮罩参数
4. 更新会话占位符映射
5. 发送遮罩后的请求到 LLM
6. 接收工具结果
7. 还原工具结果中的 PII
8. 返回还原后的结果
```

### 5. 占位符管理器（Placeholder Manager）

**职责**：
- 生成确定性占位符（基于 PII 值哈希）
- 维护占位符到原始值的映射
- 跨请求保持占位符一致性（同一值永远生成同一占位符）
- FIFO 淘汰机制

**占位符格式**（V0.9.1+）：
```
${TYPE_hash}
```

**示例**：
- `${ID_CARD_a1b2c3}`
- `${PHONE_d4e5f6}`
- `${EMAIL_f7e8d9}`
- `${CRED_abc123}`

**确定性生成**：
- 同一 PII 值永远生成同一占位符，不依赖 session 和跨轮映射
- 哈希长度可配置：`placeholder_hash_length`（6 或 8，默认 6）
- 可选 HMAC 密钥：`placeholder_hmac_key`（默认空，用 plain sha256）
- 向后兼容 `${CRED_N}` 旧格式

### 6. PII 处理器（PII Processor）

**职责**：
- 调用 OneAIFW API 检测 PII
- 生成占位符
- 还原占位符为原始值

**API 集成**：
```
POST /api/mask_text
请求: {"text": "我的身份证是 110101199003077777"}
响应: {"masked_text": "我的身份证是 ${ID_CARD_a1b2c3}", "maskMeta": "base64_encoded_metadata"}

POST /api/restore_text
请求: {"text": "用户 ${ID_CARD_a1b2c3} 存在", "maskMeta": "base64_encoded_metadata"}
响应: {"restored_text": "用户 110101199003077777 存在"}
```

## 安全设计

### PII 保护机制

1. **端到端加密**：代理与 LLM 之间的通信使用 HTTPS
2. **内存安全**：PII 数据仅在内存中处理，不写入磁盘
3. **会话隔离**：每个会话的 PII 映射完全隔离
4. **自动过期**：会话和映射自动过期，防止泄露

### 防御措施

1. **输入验证**：严格验证所有输入请求
2. **速率限制**：防止恶意请求
3. **审计日志**：记录所有 PII 操作（不包含实际 PII）
4. **最小权限原则**：仅使用必要的 API 权限

## 性能优化

### 缓存策略

1. **会话缓存**：会话数据存储在内存中
2. **PII 检测缓存**：相同文本的检测结果可缓存
3. **占位符映射缓存**：活跃会话的映射常驻内存

### 并发处理

1. **请求并发**：使用 goroutine 并发处理请求
2. **PII 检测并发**：批量检测提高效率
3. **流式处理**：SSE 流式响应减少延迟

### 资源管理

1. **连接池**：复用 HTTP 连接
2. **内存限制**：限制会话数量和大小
3. **优雅关闭**：处理完现有请求后再关闭

## 错误处理

### 错误分类

1. **OneAIFW 错误**：PII 检测失败
2. **LLM 错误**：LLM 服务不可用或响应错误
3. **会话错误**：会话不存在或已过期
4. **网络错误**：网络连接问题

### 错误恢复

1. **重试机制**：可配置的重试策略
2. **降级处理**：OneAIFW 不可用时拒绝请求
3. **超时控制**：所有外部调用都有超时限制

## 扩展性设计

### 插件架构

支持通过插件扩展功能：
- 自定义 PII 检测器
- 自定义占位符格式
- 自定义存储后端

### 多租户支持

设计支持多租户架构：
- 租户隔离
- 独立配置
- 独立监控

## 监控和可观测性

### 指标

1. **请求指标**：总请求数、成功率、延迟
2. **PII 指标**：检测数量、遮罩数量、还原数量
3. **会话指标**：活跃会话数、会话过期数
4. **错误指标**：错误数量、错误类型分布

### 日志

1. **结构化日志**：JSON 格式，便于解析
2. **日志级别**：DEBUG、INFO、WARN、ERROR
3. **敏感数据脱敏**：日志中不记录实际 PII

### 追踪

1. **请求追踪**：每个请求有唯一 ID
2. **链路追踪**：支持 OpenTelemetry
3. **性能分析**：记录关键操作耗时

## 未来改进

1. **自定义 PII 规则**：支持用户自定义 PII 检测规则
2. **联邦学习**：保护隐私的 PII 模型训练
3. **边缘部署**：支持在边缘设备上运行
4. **更多存储后端**：支持 PostgreSQL、MongoDB 等
5. **高可用性**：支持集群部署和故障转移
